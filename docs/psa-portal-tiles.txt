CAMPAIGN / OFFER-PROGRAM CONFIG  (found 2026-07-12 via user-provided trace, verified live)
================================================================================
This is the data we actually want. It is NOT a Lightdash tile — it's a plain PSA
SvelteKit data endpoint on psacard.com, same accessToken cookie auth as analytics.

Request URL:
  https://www.psacard.com/buyercampaignmanager/__data.json?x-sveltekit-trailing-slash=1&x-sveltekit-invalidated=001
Auth:    Cookie: accessToken=<harvested token>   (SAME token/cookie as itemized-purchases path)
Method:  GET, browser UA. Verified live from devcontainer -> HTTP 200.
Query params it honors (from "uses.search_params"): page, q, status  -> pagination + search + status filter.
Paging:  campaignsResponse.{pageNumber,pageSize,totalCount}; default pageSize 50.

Payload is SvelteKit "reference-packed": every value (incl. leaves) is an index into a
flat data[] array. Must deref recursively. Decoded shape per campaign item:

  campaignRequestId        uuid            e.g. 0554a104-a735-43d6-b181-51067c594216  (the campaign ID)
  campaignType             enum            "CATEGORY"
  campaignName             string          "Modern 10", "Crystal"  (matches adjusted_description in tiles)
  status                   enum            "PAUSED" | (presumably "ACTIVE"/others)
  category                 enum            "POKEMON"
  prepackagedSpecListIds   []              spec-list refs (empty in sample)
  buyerPricePercentClv     int (percent)   72, 70   <- the buy% / % of CLV offered
  buyBox: {
    gradeMin, gradeMax     string          "10".."10" / "1".."10"
    yearMin, yearMax       int             2024..2026 / 2002..2003
    priceMin, priceMax     int (USD)       500..3000   <- confirm cents vs dollars on ingest
    clvConfidenceMin       int             4
    buyerFlatFee           int             3
  }
  budget: { dailyBudget int(3000), dailySpecQuantityLimit int(2) }
  publishers: { allow, deny }   \
  subjects:   { allow, deny }    >  inclusion / exclusion controls (character/subject/spec lists)
  specs:      { deny }          /   ENCODING UNCERTAIN: both sample campaigns near-empty; the
                                    derefed values were small ints (0/2/5). Need a campaign with
                                    populated allow/deny lists to confirm whether these are ID
                                    arrays or counts.  <-- FOLLOW-UP / ask PSA dev team.
  createdAt, updatedAt     RFC3339 timestamps

Sample (2 campaigns live today, both PAUSED, both POKEMON/CATEGORY):
  Modern 10  buy% 72  grade 10-10  year 2024-2026  price 500-3000  clvConfMin 4  flatFee 3  dailyBudget 3000
  Crystal    buy% 70  grade 1-10   year 2002-2003  price 500-3000  clvConfMin 4  flatFee 3  dailyBudget 3000

Raw payload saved: docs/psa-campaigns-raw.json

{allow,deny} values point at integer leaves:
  Modern 10: publishers{allow:0,deny:0} subjects{allow:0,deny:0} specs{deny:2}

WRITE / SAVE ENDPOINT — FOUND & VERIFIED (2026-07-12). Write path now UNBLOCKED.
================================================================================
The mutation is a SvelteKit REMOTE FUNCTION (not __data.json, not a REST host):

  POST https://www.psacard.com/buyercampaignmanager/_app/remote/{buildHash}/updateCampaign
    e.g. .../_app/remote/1fvuqqe/updateCampaign
    ⚠️ {buildHash} ("1fvuqqe") is the SvelteKit app build id — it CHANGES on every PSA
       frontend deploy. Must be scraped fresh from the app (page HTML / __data.json), not
       hardcoded. Same harvest run that grabs the token can capture it. FOLLOW-UP: confirm
       where the hash is exposed (likely in the analytics/portal page HTML script imports).
  Auth: same accessToken cookie (browser context). Cloudflare constraint applies -> must
       run from the Cloudflare-trusted harvester/out-of-process context, NOT the main app.

Request body (application/json):
  { "payload": "<base64>", "refreshes": [] }
  payload = base64 of a SvelteKit reference-packed array. Decoded arg[0]:
    { "id": "<campaignId>", "formData": { ...FULL formData... } }
  >>> The FULL formData is echoed (not a partial patch) — SAME shape as the edit endpoint's
      formData (campaignName, campaignType, category, prepackagedSpecListIds, isActive,
      bidPercentage, flatFee, dailyBudget, dailySpecLimit, grade/year/price min/max,
      cardLadderConfidenceMinimum, publisherFilterType, selectedPublishers[],
      subjectFilterType, selectedSubjects[]{id,name}, deniedSpecs[]).
  => WRITE STRATEGY: GET edit endpoint -> deref formData -> mutate the field(s) we approve ->
     re-pack to reference-packed base64 -> POST. Read-modify-write of the whole record.
     (Mirrors our internal hard rule: PUT full campaign record, never PATCH.)

Reference-packing for the REQUEST (we must ENCODE, not just decode):
  arg is serialized as a flat array where every value (incl. nested) is replaced by its
  index in the array; index 0 is the root. Need an ENCODER (dedup-by-value like devalue).
  NOTE observed: PSA's own encoder de-dupes scalars (priceMaximum reused index 10 = 3000,
  same as dailyBudget). Our encoder does not need to match byte-for-byte — server derefs by
  index — but must produce a valid self-consistent packed array. VERIFY with a live round-trip.

Response: { "type":"result", "result":"<json-string of a reference-packed array>" }
  Decodes to the updated campaign (list-row shape + an "audit" block:
  {buyerAccountId, createdByCid, updatedByCid}). Use to confirm the write.

VERIFIED LIVE: user's in-browser Save changed Crystal bidPercentage 70->72; re-reading the
  list endpoint shows buyerPricePercentClv=72, updatedAt=2026-07-12T21:02:44 (== save resp).
